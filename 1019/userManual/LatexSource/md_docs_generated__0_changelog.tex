\doxysection*{Known problems}


\begin{DoxyItemize}
\item If the C\+U\+DA support is enabled, then the \mbox{\hyperlink{group__starneig__sm__sep_ga3180b21861b10ebf4016ee15e2f79d82}{starneig\+\_\+\+S\+E\+P\+\_\+\+S\+M\+\_\+\+Hessenberg()}} interface function cannot always handle problems that do not fit into G\+PU\textquotesingle{}s memory. The cause of this problem is not known.
\item The outputs of the \mbox{\hyperlink{group__starneig__sm__gep_gaaf1e3ec2fc5d8cfb7a7a5464d23a25ed}{starneig\+\_\+\+G\+E\+P\+\_\+\+S\+M\+\_\+\+Schur()}} and \mbox{\hyperlink{group__starneig__dm__gep_ga3ffea6beaaec769b16b8f49cb8809a4d}{starneig\+\_\+\+G\+E\+P\+\_\+\+D\+M\+\_\+\+Schur()}} interface functions are not always in the so-\/called standard format. It is possible that some diagonal entries in the right-\/hand side output matrix are negative.
\item The \mbox{\hyperlink{group__starneig__sm__gep_ga74739cbe0c292aaaf7c28c9fd383ddcb}{starneig\+\_\+\+G\+E\+P\+\_\+\+S\+M\+\_\+\+Eigenvectors()}} interface function may scale the input matrices.
\end{DoxyItemize}

\doxysubsection*{Known compatibility problems}

\doxysubsubsection*{B\+L\+AS}


\begin{DoxyItemize}
\item With some Open\+B\+L\+AS versions, it is necessary to set the {\ttfamily O\+P\+E\+N\+B\+L\+A\+S\+\_\+\+N\+U\+M\+\_\+\+T\+H\+R\+E\+A\+DS} environmental variable to value {\ttfamily 1} ({\ttfamily export O\+P\+E\+N\+B\+L\+A\+S\+\_\+\+N\+U\+M\+\_\+\+T\+H\+R\+E\+A\+DS=1}).
\item Some M\+KL versions can cause poor scalability. The problem appears to be related to Intel\textquotesingle{}s Open\+MP library. Setting the {\ttfamily K\+M\+P\+\_\+\+A\+F\+F\+I\+N\+I\+TY} environmental variable to value {\ttfamily disabled} fixes the problem ({\ttfamily export K\+M\+P\+\_\+\+A\+F\+F\+I\+N\+I\+TY=disabled}).
\item Open\+B\+L\+AS version 0.\+3.\+1 has a bug that can cause an incorrect result.
\item Open\+B\+L\+AS versions 0.\+3.\+3-\/0.\+3.\+5 can cause poor scalability.
\end{DoxyItemize}

\doxysubsubsection*{M\+PI}


\begin{DoxyItemize}
\item Some older Open\+M\+PI versions ($<$= 2.\+1.\+1) have a bug that can cause a segmentation fault during a parallel A\+ED.
\item The library has an unsolved memory leak problem with Open\+M\+PI. Only large problem sizes are effected. It is not known whether this problem is related to Star\+N\+Eig, Star\+PU, Open\+M\+PI or something else. The problem is known to occur with P\+M\+Ix 2.\+2.\+1, U\+CX 1.\+5.\+0, Open\+M\+PI 3.\+1.\+3, and Star\+PU 1.\+2.\+8. The memory leak is sometimes accompanied by the following warning\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{mpool.c:38   UCX  WARN  object 0x2652000 was not returned to mpool ucp\_requests}
\end{DoxyCode}

\item The test program can trigger the following bug in U\+CX 1.\+6.\+1\+: \href{https://github.com/openucx/ucx/issues/4525}{\texttt{ https\+://github.\+com/openucx/ucx/issues/4525}}
\end{DoxyItemize}

\doxysubsubsection*{Star\+PU}


\begin{DoxyItemize}
\item For optimal C\+U\+DA performance, Star\+PU version that is newer than 1.\+3.\+3 is recommended.
\item Star\+PU versions 1.\+2.\+4 -\/ 1.\+2.\+8 and some Star\+PU 1.\+3 snapshots cav cause poor C\+U\+DA performance. The problem can be fixed by compiling Star\+PU with {\ttfamily -\/-\/disable-\/cuda-\/memcpy-\/peer}. It is possible that newer versions of Star\+PU are also effected by this problem.
\item The {\ttfamily S\+T\+A\+R\+P\+U\+\_\+\+M\+I\+N\+I\+M\+U\+M\+\_\+\+A\+V\+A\+I\+L\+A\+B\+L\+E\+\_\+\+M\+EM} and {\ttfamily S\+T\+A\+R\+P\+U\+\_\+\+T\+A\+R\+G\+E\+T\+\_\+\+A\+V\+A\+I\+L\+A\+B\+L\+E\+\_\+\+M\+EM} environmental variables can be used to fix some G\+P\+U-\/related memory allocation problems\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{STARPU\_MINIMUM\_AVAILABLE\_MEM=10 STARPU\_TARGET\_AVAILABLE\_MEM=15 ...}
\end{DoxyCode}

\end{DoxyItemize}
